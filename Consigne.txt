RAG Pipeline Project for Foreign Students – Documentation

1. Overview
This project aims to build a RAG (Retrieval-Augmented Generation) pipeline for foreign student documents (legal, visa, immigration, aid, etc.). The focus is on per-document processing, metadata management, JSONL generation, and Google Drive integration.
Raw documents are stored on Google Drive.


Processing generates per-document JSONL chunks.


Embeddings and Vector DB ingestion will be handled later.


Project goals:
- Maintain a clean folder architecture in Google Drive for raw documents, JSONL chunks, and metadata.
- Automate the pipeline from document collection → chunking → metadata addition → JSONL creation.
- Use subagents efficiently to parallelize and optimize tasks, without unnecessary imports.
- Allow per-document processing via metadata JSON lines (one line per document).


2. Google Drive Folder Structure
GDrive/
└─ DB/
   ├─ pdfs/           # Raw PDF files
   ├─ html/           # Raw HTML files if any
   ├─ jsonl/          # Per-document JSONL chunks ready for embedding
   └─ meta/           # Metadata JSON/CSV linking documents to source URLs, Drive links, category, date, jurisdiction, authority_score

Each document has a unique document_id.


JSONL chunks are stored in GDrive/DB/jsonl/<document_id>.jsonl.


Status tracking per document:


{
    "jsonl_ready": true,
    "embedding_done": false
}



3. Metadata Format
Each line/object in metadata.json or CSV includes:
{
  "document_id": "pref92_visa2025",
  "document_type": "pdf",         // "html" or "url"
  "drive_link": "https://drive.google.com/...",
  "source_url": "https://prefecture92.gouv.fr/visa-renewal",
  "category": "titre_sejour",
  "sub_category": "delai_renouvellement",
  "jurisdiction": "Hauts-de-Seine",
  "date": "2025-01-10",
  "authority_score": 3,
  "language": "fr"
}

Notes:
document_type indicates how the pipeline reads the document.


drive_link allows automatic download from Drive.


Metadata fields will be attached to each JSONL chunk (chunk_id, page_start, page_end, chunk_size).

Fetch / Read Document


Document_Type

Text Extraction & Preprocessing


Remove boilerplate (headers, footers, repeated info)


Normalize whitespace, remove special characters if needed


Handle multiple languages (detect and tag)


Optionally OCR PDFs if text not selectable (pytesseract + pdf2image)


Cleaning / Preprocessing


Remove irrelevant sections (ads, navigation, menus)


Standardize date formats


Normalize punctuation and encoding


Remove duplicated paragraphs across pages


Optional: lowercase, strip HTML tags, remove scripts/styles


Chunking


Split text into chunks of ~500 words (configurable and logic chunks Try to split along logical sections (paragraphs, headers) rather than mid-sentence.)


Align with logical sections (do not cut mid-sentence or mid-paragraph)


Assign chunk_id, page_start, page_end, chunk_size metadata
Include ~50 words overlap between consecutive chunks. Why: ensures context continuity, so the LLM doesn’t miss information that spans across chunk boundaries.


Metadata Attachment


Each chunk inherits all document metadata:


document_id, category, sub_category, jurisdiction, source_url, drive_link, language, authority_score


Add chunk-specific fields:
Metadata per chunk (must-have for performant RAG)
document_id → to link chunk to source


chunk_id → unique identifier per chunk


text → chunk text


category / sub_category → topic of the document


jurisdiction → regional info


source_url / drive_link → origin of document


date → document date


authority_score → weight of reliability


language → language of text


page_start / page_end → PDF page info


chunk_size → number of words in chunk


Optional: section_header or heading if text contains structured sections


This allows vector search + filtering by category, authority, language, etc., for high-quality RAG retrieval.


chunk_id, page_start, page_end, chunk_size

Other Considerations for Ultra-Performant RAG
Vector DB metadata → include all chunk metadata as fields for filtering and relevance boosting.


Reranking → optionally rerank results from similarity search based on authority_score, category, or recency.


Chunk-level indexing → ensures retrieval granularity and prevents LLM hallucinations.


Language-specific embeddings → if docs are multilingual, make sure embedding model handles all languages.


Traceability → keep drive_link and page_start for audit and referencing.


Incremental updates → new docs produce new JSONL chunks without recomputing existing ones.




4. **Subagent usage**:
   - Use the most relevant subagents to parallelize tasks:
       - `data-engineer`, `nlp-engineer`, `python-pro` → pipeline automation
       - `database-administrator`, `postgres-pro` → Vector DB handling
       - `workflow-orchestrator`, `task-distributor`, `multi-agent-coordinator` → parallelization
       - `documentation-engineer` → README/usage generation
       - Optional: `legal-advisor` → verify official/legal documents
   - Do **not import unnecessary subagents**
   - Explicitly notify whenever a subagent is invoked:  
     Example: "🔧 Using subagent: python-pro → Reason: parsing PDF and cleaning text"

5. **Project-level subagent registration**:
   - Import relevant subagents from global installation (`~/.claude/agents/`) into `.claude/agents/` for project scope
   - Register them to automatically trigger for relevant tasks

Validate chunks in JSONL.
